Generated by /office-hours on 2026-06-17. Status: DRAFT (pending approval → /plan → build).

### What "universal / any agent" honestly means here (scope truth)
You chose **Claude Code native** as the substrate, accepting the portability tradeoff. So universality is delivered in **two honest tiers, not by a multi-backend runtime**:
- **Tier 1 — deep, CC-native:** hooks (SessionStart/PreToolUse/PostToolUse), the richest enforcement + orchestration UX. Best experience, Claude-Code-only.
- **Tier 2 — portable to any agent:** every engine package (compress, memory, web, guard's policy/sandbox/skill-vetting/drift-guard) ships as **a standalone MCP server + CLI**. Any MCP-capable host (Cursor, Cline, Codex, Gemini, OpenAI-Agents) gets the *tools and the enforcement primitives*; skills port via multi-host SKILL.md generation.
- **What does NOT travel:** the hook-driven *automatic* enforcement (a non-CC host must call the policy/sandbox/vet CLIs itself). We state this rather than pretend the floor is free everywhere.

This is why we keep omnigent's **policy-engine + sandbox as host-portable standalone processes** (Tier 2) but still **drop its multi-backend runtime chassis** (we don't host other agents' loops). The crown-jewel enforcement floor is portable as *primitives*, automatic only under CC.

## 1. Problem Statement

Every "best harness" claim on the market is a partial. Skill collections have no enforcement floor. Memory tools don't compress. Compression tools don't remember. Orchestration frameworks assume you ARE Claude Code. Security is bolted on, not below the model. The user runs a messy, overlapping pile today (gstack + caveman + claude-mem + graphify + ~700 connector skills + browse) where every layer exists but nothing is rationalized.

Goal: one coherent, clean-room, Claude-Code-native plugin monorepo that implements **every harness layer, best-of-breed, with zero feature overlap**, publishable as the reference harness.

## 2. What a Harness Actually Is (the rubric)

A harness is the **control-plane OS around a stateless token-predictor**. The model is probabilistic reasoning; the harness turns that into deterministic, dependable, *safe* action. Per the landscape research, four pillars:

1. **Context management** — system prompt, history, retrieval, compaction.
2. **State & memory** — scratchpads, task lists, short/long-term.
3. **Control flow** — retries, budgets, stop conditions, sub-agent delegation.
4. **Safety & observability** — guardrails, permissions, approval gates, tracing.

Compliance targets the design must hit:
- **12-Factor Agents** (esp. #3 *own the context window*, #4 *tools are structured outputs*, #7 *human contact is a tool call*, #8 *own the control flow*, #9 *compress errors into context*, #10 *small focused agents*).
- **OWASP LLM06:2025 Excessive Agency** (structured authz, not prompt-level trust).
- **Anthropic** context-engineering + long-running-harness + writing-tools guidance.
- **Enforce below the LLM** — every safety/scope rule is deterministic code (hook/gate/linter), never a polite request in a prompt.

## 5. The Ideal Harness Architecture (9 layers, anti-overlap enforced)

Each capability has **exactly one** chosen mechanism. Losers are named so nothing duplicates.

**L0 — Substrate / packaging.** Monorepo → CC plugin marketplace. `marketplace.json` + per-module `plugin.json` (addyosmani/pm-skills schema). Skill templating + multi-host generation, Claude/Codex/Gemini/Cursor (gstack `gen-skill-docs` + Superpowers/gsd-core multi-host). `validate_plugins.py` lint in CI. `references/` progressive disclosure (K-Dense). SessionStart injects a `using-ideal-harness` bootstrap skill (Superpowers/addyosmani). **MCP-server hosting (Tier-2 universality):** every engine package (L1/L2/L3/L6) exposes a standalone MCP server + CLI so any MCP-capable host gets the tools and enforcement primitives, not just Claude Code.

**L1 — Context & Compression.** *Single compression module, four non-overlapping tactics:*
- input-side: headroom deterministic `tool_result` compression + CCR retrieval tool (MCP).
- output-side: caveman terse mode (toggle).
- server-side: respect `cache_control` frozen-floor (never recompress cached prefix).
- offload: *trigger* a flush to L2 memory (retrieve-not-recall). L1 only triggers; **L2 owns the store** (no duplication).
- error-compression (12-factor #9): collapse stack traces / repeated tool errors into a one-line cause + count before they re-enter context.
- Emulate Claude Code's progressive compaction + pre-compaction flush to memory file. *(LMCache rejected — wrong layer.)*
- **L1 owns NO safety control.** Secret redaction lives in L6 (always-on), never gated behind a compression toggle.

**L2 — Memory.** *One module, two stores, no overlap:*
- structural: graphify code-graph + token-budgeted subgraph retrieval (MCP).
- episodic: claude-mem hook→XML-observation→SQLite-FTS5, retrieved by **BM25 (native FTS5) + int8-vector hybrid, fused with RRF** (claude-mem ships FTS5 but sorts by recency; we use the rank it already has + add a vector rerank — int8 idea take-parts'd from codebase-memory-mcp). hermes `curator` maintains it (reconcile claims vs tool-call evidence).
- **L2 owns context-overflow persistence**: when L1 triggers a flush, the bytes land here, not in two places. *(ruflo memory-bridge rejected — graphify+FTS5 superior.)*

**L3 — Web & Research.** *One module, three sub-tools by axis (the adjudicated boundaries):*
- interactive automation/debugging: chrome-devtools-mcp pattern (CDP + uid snapshot) via a gstack-/browse-style warm-Chromium daemon (atomic state, health auto-start, idle shutdown, ONNX/Haiku injection classifier).
- extraction (URL→markdown): firecrawl idea clean-room (engine-cascade + SSRF guard + HTML→md). No SaaS.
- multi-source research: last30days fusion/rerank/cluster/HTTP engine + connectors; Agent-Reach channel-routing + `doctor` health-check. *(claude-in-chrome rejected — closed/locked.)*

**L4 — Skills library.** SDLC skills clean-room (addyosmani): spec-driven-dev, TDD, code-review, debugging, context-engineering. writing-skills = TDD-for-prompts (Superpowers). Anti-rationalization tables + binary verification checklists baked into each.

**L5 — Orchestration & control flow.** subagent-driven-development (Superpowers): controller + fresh-context-per-task + spec/quality review-gate + fix loop + **file-based artifact handoff** + **durable git-path ledger** (survives compaction). autoplan dual-model consensus gauntlet (gstack) for planning/review with the 6-principle auto-decision engine. brainstorming HARD-GATE. hermes self-registering tool registry + SHA-256 loop/no-progress guardrails. **Control-flow capabilities its own rubric (§2 pillar 3) demands, now explicit:** (a) **spend governor** — token+dollar accounting, per-run hard cap, graceful abort (mirrors Voraxx's spending-cap safeguard); (b) **API-level retry/backoff/circuit-breaker** for Anthropic 429/overloaded/stream-reconnect — DISTINCT from L3's web-fetch retry; (c) **session resume/pause/checkpoint** — crash-recovery from a checkpoint, not just the artifact ledger; (d) **HITL escalation** — clarification/approval as a first-class tool call (12-factor #7), routed to L6's ask-gate. *(ruflo "swarm" rejected — fake.)*

**L6 — Security & Enforcement floor (below the LLM, non-negotiable).** Anthropic checklist: permissions allow/ask/deny (deny wins, fail-closed); `denyRead` on `~/.aws`, `~/.ssh`; self-policy write-protection; egress allowlist (prompt-on-first-use, curl/wget not auto-approved); subprocess env-scrub; OS sandbox (Seatbelt/bubblewrap). omnigent CEL policy engine + sandbox. gsd-core drift-guard authority ladder (verify symbols, hard-block hallucinations). hermes `<untrusted_tool_result>` injection wrapping. SkillSpector vetting gate (scan every third-party skill before load: 64-pattern taxonomy + AST + taint + OSV + homoglyph). gstack deterministic PreToolUse hooks. OWASP LLM06 framing. **HITL = the ask-gate IS 12-factor #7** (clarification/approval as a tool call; L5 routes escalations here). **Secrets broker** — positive, scoped, short-lived credential injection so a tool gets only the secret it needs, never landing in context/logs (distinct from deny-read; mirrors Voraxx's per-compartment secret namespaces). **Always-on secret-redaction hook** (relocated out of L1/caveman — a safety control must never depend on a compression toggle).

**L7 — Design & Taste.** impeccable `detect.mjs` anti-pattern linter wired to PostToolUse hook + reflex-reject lists + slop test + per-model defect blocks. taste-skill three dials + brief→design-system routing. design-extract WCAG/contrast/drift audit (extraction feeding taste). emilkowal.ski animation-review framing. *(ECC rejected.)*

**L8 — Eval & Observability.** feature-list pass/fail JSON gate; mandatory e2e/browser verification (Anthropic long-running); LLM-eval harnesses (gstack pattern); OTel tracing + hash-chained audit ledger; `init.sh` + standard session choreography; one-feature-per-session discipline.

Each `packages/*` is an independently installable plugin; `core` is required; others opt-in. This is the boil-the-lake full spine but modular.

## 8. Approaches Considered

**APPROACH A — Modular plugin monorepo (RECOMMENDED).** 9 packages, each independently installable, `core` required. Effort: XL. Risk: Med. Pros: full spine, no overlap, sellable module-by-module, mirrors Voraxx's own enterprise-module philosophy. Cons: large surface. Reuses: every verdict above. Completeness 10/10.

**APPROACH B — Single mega-plugin.** One plugin, all layers inline. Effort: L. Risk: Med-High. Pros: simpler install. Cons: can't adopt piecemeal, harder to test, violates "small focused" 12-factor. Completeness 7/10.

**APPROACH C — Meta-installer over upstream.** Curate/pull upstream. Effort: M. Rejected at decision time (user chose clean-room; inherits AGPL/churn). Completeness 5/10.

**RECOMMENDATION: A** — modular monorepo. Matches the locked decisions (CC-native, clean-room, full-spine) and the no-overlap requirement, and lets each layer be tested and shipped independently.

## 9. Distribution Plan

- Publish as a Claude Code plugin marketplace (`marketplace.json`) installable via `/plugin marketplace add` + per-module `/plugin install`.
- npm for any CLI sidecars (browse daemon, compress proxy) under one scope.
- Multi-host SKILL.md generation so Codex/Gemini/Cursor consume the same skills (this IS the "universal/any-agent" story for a CC-native build).

## 10. Risks / Open Questions

- **R1 Scope.** Full spine is XL. Mitigate: ship `core + guard + compress + memory + orchestrate` as **v0.1** (a real harness — it can *plan, execute, remember, compress, and be governed*). `web + skills + design + eval` as **v0.2**. (Swapped `web`→`orchestrate` vs first draft: memory+compress+guard with no orchestration is a library, not a harness; and web's hardest piece is deferred anyway.) Still full-spine over two tight releases.
- **R2 Clean-room fidelity.** Several "packages" are sub-system-sized. Ranked by reimplement risk: **(1) SkillSpector L6 vetting is the biggest** — AST + taint + OSV + homoglyph across arbitrary skills is a security product. Mitigate: shell out to existing OSV + semgrep, reimplement only the 64-pattern signature DB + homoglyph check; do NOT hand-roll taint analysis. (2) firecrawl **SSRF guard** is security-sensitive — clean-room SSRF guards classically miss DNS-rebinding / redirect-following / IPv6+decimal-IP bypasses; write it with those tests first. (3) headroom byte-range surgery, graphify dedup, gsd-core authority ladder — lift the documented algorithm, fresh code + own tests. (4) L8 = three systems (OTel + audit-ledger + eval-harness); L3 research-fusion is itself a project. Don't let "9 packages" hide this.
- **R3 Browser engineering.** A warm-Chromium daemon is the single biggest build (gstack's is ~24K LOC) — **deferred to v0.2**. v0.2 `web` interactive sub-tool depends on chrome-devtools-mcp as an external MCP first, reimplements the daemon later. v0.2 `web` extraction+research sub-tools (firecrawl-idea scraper + last30days fusion) have no daemon dependency.
- **R4 Name/location.** Codename The Ideal Harness; repo at `/Users/Bharat/Voraxx/Harness` as standalone git repo (add to Voraxx `.gitignore`). Confirm at build.
- **Q1** License: MIT vs Apache-2.0 (patent grant)?
- **Q2** Python vs TypeScript for the engine packages (graphify/last30days are Python; headroom is Rust+Python; gstack/gsd-core are TS)? Likely polyglot per package.
